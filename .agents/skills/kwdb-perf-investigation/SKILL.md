# KWDB Performance Investigation

Thin orchestration layer. Contains NO TSBS logic and NO perf logic — only routes
user intent and coordinates multi-skill pipelines.

## Scope

Only TSBS-based performance analysis is supported. All pipelines go through
`tsbs-benchmark` as the workload driver. Non-TSBS perf requests should be
declined with an explanation of this limitation.

## Pipeline Patterns

### Pattern 1: Standard TSBS Benchmark

```
User: "测一下场景二的TSBS性能"
  → tsbs-benchmark (Entry A)
```

No orchestration needed. Dispatch directly to `tsbs-benchmark` with the user's
parameters. tsbs-benchmark handles the full pipeline end-to-end.

### Pattern 2: Perf Profiling of TSBS Workload

```
User: "用perf分析场景一high-cpu-all的查询瓶颈"
  │
  ├─→ Step 0: Create session directory
  │     SESSION_DIR=<repo>/tsbs-profile/2026-06-02_14-30-00/
  │     Persist metadata: commit, scene, query type, trace target → session_metadata.txt
  │     Persist uncommitted changes → git.diff
  │
  ├─→ Step 1: Skill(tsbs-benchmark, Entry B)
  │     generate_wrapper.sh 100 load high-cpu-all -o $SESSION_DIR
  │     Start KWDB → generate wrapper → output Setup Report
  │
  ├─→ Step 2: Pre-load data (query only)
  │     Bash: source $SESSION_DIR/tsbs_workload.sh && do_load
  │     Skip this step when tracing load.
  │
  ├─→ Step 3: Skill(perf-event-profiling, Profiling mode)
  │     Generate $SESSION_DIR/tsbs_perf_runner.sh (see below)
  │     Call perf skill with PID + runner script path → perf.data
  │
  └─→ Step 4: Skill(perf-event-profiling, Analysis mode)
        Hot functions → metrics → source-level analysis
        Full report (including per-function hotspot drilldown)
        → $SESSION_DIR/analysis_report.md
```

Five-stage pipeline:

0. Create session directory `<repo>/tsbs-profile/<YYYY-MM-DD_HH-MM-SS>/`.
   Persist metadata and code state:
   - `$SESSION_DIR/session_metadata.txt` — commit ID, scale, tsbs_case, query type(s),
     trace target (load/query), timestamp
   - `git diff HEAD > $SESSION_DIR/git.diff` — uncommitted changes (if any)
1. Call `tsbs-benchmark` (Entry B). Always pass `load` to `generate_wrapper.sh`
   so the wrapper includes `do_load`. Wait for the Setup Report.
   **Timeout:** data generation may take >5 minutes for large scales; set a
   generous timeout (e.g. 10 minutes) on this step.
2. **Query only:** Execute `do_load` directly (Bash tool) to populate data before
   profiling. **Load only:** Skip this step — `do_load` is the trace target itself.
3. Generate `$SESSION_DIR/tsbs_perf_runner.sh` (see below). Call
   `perf-event-profiling` in Profiling mode: pass PID and runner script path.
   The perf skill runs capture via `perf-capture.sh` and produces
   `$SESSION_DIR/perf.data`.
4. Call `perf-event-profiling` in Analysis mode: pass `$SESSION_DIR/perf.data`.
   The analysis MUST include:
   - Flame graphs (`$SESSION_DIR/flamegraph.svg` + `$SESSION_DIR/flamegraphs/`)
   - Top hot functions and key metrics (IPC, cache miss rate)
   - **Source-level analysis** (DSO identification → debug info check →
     per-instruction hotspot drilldown → addr2line source mapping → source
     code correlation → per-function hotspot report with root cause and
     suggestions)
   Write the full report to `$SESSION_DIR/analysis_report.md`.

### Session Directory

```
<repo>/tsbs-profile/2026-06-02_14-30-00/
  ├── session_metadata.txt      # commit ID, scale, case, query type(s), trace target, timestamp
  ├── git.diff                  # git diff HEAD — uncommitted changes, if any
  ├── tsbs_workload.sh          # generated by generate_wrapper.sh
  ├── tsbs_perf_runner.sh       # trace target, fed to perf skill (Profiling mode)
  ├── perf.data                 # perf capture output
  ├── flamegraph.svg            # primary flame graph (cycles)
  ├── flamegraphs/              # per-event flame graphs (cache-misses, branch-misses, etc.)
  └── analysis_report.md        # perf analysis report (hot functions, metrics, root cause)
```

Example `session_metadata.txt`:
```
commit=419bae3a23b23377df2471667a57b16a0f7d8089
scale=4000
case=cpu-only
query_types=high-cpu-all
trace_target=query
timestamp=2026-06-02_14-30-00
```

### Runner Script

The orchestrator generates `$SESSION_DIR/tsbs_perf_runner.sh` containing only the
trace target. Passed to `perf-event-profiling` (Profiling mode) as the `-c` argument.
Load is never looped; query loops for 30s.

**Trace load:**
```bash
#!/bin/bash
source $SESSION_DIR/tsbs_workload.sh
do_load
```

**Trace query:**
```bash
#!/bin/bash
source $SESSION_DIR/tsbs_workload.sh
start_time=$(date +%s)
while true; do
    do_query >/dev/null 2>&1
    if [ $(($(date +%s) - start_time)) -ge 30 ]; then
        break
    fi
done
```

## Contract

### Setup Report

tsbs-benchmark (Entry B) outputs a markdown Setup Report upon successful setup.
The orchestrator consumes it to extract:

- **KWDB PID**
- **Wrapper path** — `$SESSION_DIR/tsbs_workload.sh`
- **Available functions** — `do_load` and/or `do_query`

Each session has its own directory (`tsbs-profile/<timestamp>/`). Concurrent
profiling sessions are not supported — perf itself would interfere, so this is
by design.

### PID File

The PID file path is a known constant: `tsbs-benchmark/data/tsbs_n1/kwbase.pid`.
The orchestrator uses this path for cleanup regardless of whether tsbs-benchmark
succeeded or failed.

## Error Handling

When tsbs-benchmark (Entry B) fails:

1. Read the known PID file. If a PID exists, kill KWDB.
2. Translate the raw error into user-facing guidance (e.g., "wrapper generation
   failed — check TSBS toolchain") rather than passing through raw output.

No timeouts are set on pipeline steps. Data generation and loading can be
long-running; the orchestrator waits for each step to finish naturally.

## Routing

| User mentions | Route |
|---------------|-------|
| TSBS benchmark keywords: `TSBS`, `tsbs`, `scale`, `cpu-only`, `iot`, `场景`, `写入`, `查询`, `load`, `query`, `benchmark` (as TSBS workload), without perf | → `tsbs-benchmark` (Entry A) |
| Perf keywords: `perf`, `profile`, `火焰图`, `flame graph`, `热点`, `瓶颈`, `定位`, with TSBS context | → Pipeline: `tsbs-benchmark` (Entry B) + `perf-event-profiling` |
| Perf keywords without TSBS context, or non-TSBS performance requests | → Decline: explain that only TSBS-based analysis is currently supported |

When routing to `tsbs-benchmark`, pass through all known context (scale, query
type, target operation, commit/branch). tsbs-benchmark handles its own parameter
confirmation.

## Output

After orchestration completes, summarize:

1. Which pipeline was used
2. Results from each skill in the pipeline
3. Any unresolved items
