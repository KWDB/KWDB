# 人机协作模型

本文为 [`ai-collaboration-model.md`](ai-collaboration-model.md) 的中文对照稿。

本文为仓库推荐的「多人 + AI 辅助」交付方式。设计上以工件为驱动：详尽需求正文留在仓库外，仓库保存稳定执行规则、可重复技能条目与验证指引。

## 核心原则

- 人类所有者对设定目标、划界、兼容性决策与授权发布负责。
- AI 的输出应可被评审的材料，而不只是会话回答。
- 小批交付：收窄任务范围、收窄 diff、收窄验证面。
- 验证与实现同等地位，不能在代码生成后再随便补缀。

## 工件模型

对较大规模工作跟踪下列工件：

| 工件 | 存放位置 | 责任人 | 作用 |
| --- | --- | --- | --- |
| 需求或缺陷记录 | 外部系统 | 人类负责人 | 说明工作为何存在——权威上下文 |
| 设计或拆解 | 外部系统或工单 | 负责人（可人机协作） | 定义范围、不变量与验收 |
| 代码与测试 | 仓库分支 | 实现者(agent)+评审 | 可执行变更 |
| 验证证据 | PR、CI 日志、基准结果 | 实现者(agent) | 证明主张成立 |
| 发布记录 | 发布跟踪系统或 PR | 发布所有者 | 上线与回溯轨迹 |

## 交付阶段

| 阶段 | 必备输入 | AI 产出 | 离场门禁 | 示例度量 |
| --- | --- | --- | --- | --- |
| 接收 | 外部 issue、缺陷或诉求 | 结构化摘要、风险标签、通道选择 | 已命名目标、范围、不变量、验证方案 | 接收周期、澄清轮次 |
| 设计 | 已批准的需求 + 仓库上下文 | 影响分析、任务拆分、测试策略 | 假设显式写出，所有者认可边界 | 设计提前量、返工率 |
| 实现 | 已收敛任务与受影响模块图 | 代码、测试、迁移说明、PR 草稿 | 本地构建与聚焦测试通过 | 首 PR 时间、AI「完成」率 |
| 评审 | PR、diff、CI | 风险摘要、修复建议、变更说明 | 评审者能复述原因、内容与验证依据 | 评审周期、评论命中率 |
| 验证 | 分支或候选构建 | 定向回归汇报、性能材料 | 失败路径与最近未受影响路径均已检查 | 验证成本占比、逃逸缺陷率 |
| 发布 | 已批准变更与上线方案 | release 清单、smoke、回滚备忘 | 上线与回滚路径双明确 | 发布频率、变更失败率 |
| 发布后 | 线上指标、日志、事件 | 异常摘要与后续工单 | issue 结案或拆分后续项 | MTTR、回滚占比 |

## 执行通道（如何选择）

在满足质量前提下选最轻的通道：

- **Ask**：小范围事实求证、单次查仓库、一次性命令。
- **Single-agent**：单块代码变更带聚焦验证。
- **Leader + specialist**：跨模块、性能敏感或发布敏感，需要并行拆解/实现/验证时。

应升级通道当：

- 任务跨越模块；
- 失败非确定性或性能敏感；
- diff 单次难以读完。

## 阶段自检摘要

### 接收（Intake）

- 是否记下外部需求的链接？
- 任务归类为缺陷、特性、重构还是性能？
- 回滚预期是否已知？

### 实现

- diff 是否落在承诺范围之内？
- 测试是否覆盖「改动路径」与「最近未被破坏路径」？
- agent 无法验证的假设是否写明？

### 评审与验证

- 评审者能否根据记录复现所用的精确命令？
- 日志外形、兼容性、性能风险是否点明？
- CI 失败后是否在重跑前完成分类？

### 发布

- 是否有冒烟检查？
- 是否有触发回滚的条件与动作？
- 面向运维/用户的可见改动是否记入文档？

## 生产力度量

避免 LOC、提交数等虚荣指标；宜关注：

- `lead time`：任务接受到合并；
- `time to first PR`：开始到首张可评审 diff；
- `AI merge rate`：AI 辅助 PR 合入占比；
- `verification tax`：评审+验证时间 / 实现时间；
- `change fail rate`：需回滚、热修或事故的发布占比；
- `escaped defect rate`：合并后或线上才发现的缺陷。

解读提示：

- 吞吐高但若验证税负也高，只是把负担挪移而非消除。
- AI 合入率高但变更失败率升，多半是门禁偏弱。
- 目标应是低逃逸缺陷下的合理前置时间，而非单纯堆砌代码体积。
