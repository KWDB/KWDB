# Pull Request 指南

本文为 [`pr-guide.md`](pr-guide.md) 的中文对照稿。

本文为 agent 与人类贡献者编写的变更定义最低评审质量标准。

## 仓库背景

- 仓库同时包含 Go 与 C/C++ 路径。
- 变更常影响 SQL 语义、时序行为、共享日志、共享错误路径或回归脚手架。
- `kwbase/CONTRIBUTING.md` 指向 Gitee 上公开的 KaiwuDB 上游贡献流程。**本文件只做仓库侧的补充质量标准，不替代上游流程。**

## 标题

- 具体指明真实改动范围。
- 需要时可带模块前缀，例如 `kwbase:`、`kwdbts2:`、`common:`、`qa:`。
- 若关联已跟踪缺陷/需求，即使标题短小，正文也应包含 issue 标识。

## 正文应包含

- 背景或问题陈述；
- 实际代码改动的概要；
- 已做验证及相关**具体命令**；
- 兼容性、 rollout 或运维可见说明（如适用）；
- 已知风险、缺口或后续项。

## PR 正文格式

本文件同时作为仓库内维护的 PR 正文结构来源。

建议正文包含以下 section：

- `## Summary`
- `## Verification`
- `## Risks`
- `## AI / KB Usage`

其中 `## AI / KB Usage` 是给 CI 消费的 PR 级披露区块，字段格式为：

- `AI-Assist: yes|no`
- `AI-Model: ...`
- `AI-Agent-Tool: ...`
- `AI-Session-Count: ...`
- `KB-Docs-Used: ...`
- `KB-Missed: yes|no`
- `KB-Missed-Note: ...`

请直接在 PR 正文中维护该区块，确保 CI 解析器始终读取一致的字段名。

## 链接期望

- 有关联 issue / 缺陷 / 需求时给出链接；
- 跨模块或复杂变更应链接权威的**外部**规格与设计材料；
- 性能敏感改动附基准或回归对比材料；
- 仅当确有实质帮助时再贴日志、错误片段或截图。

## 评审期望

- diff 尽量小、可读；
- 纯重构与语义变更能拆则拆；
- 日志、错误码、兼容性、基准影响须在描述或评论中明示；
- 禁止用含糊总结掩盖大范围行为变更。

## 给 Agent 的提示

- 披露假设；
- 披露未能验证的领域；
- 避免过度自信措辞；
- 若因环境所限跳过验证，须在文中写清楚。
