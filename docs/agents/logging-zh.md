# 日志指引

本文为 [`logging.md`](logging.md) 的中文对照稿。

本文为在本仓库中添加或修改日志的方式约定。

## 当前实现事实

- 共享日志实现位于 `common/src/log/`。
- `lg_impl.cpp` 将 `DEBUG`、`INFO`、`WARN`、`ERROR`、`FATAL` 映射到内部严重程度。
- 严重程度展示由 `lg_severity_test.cpp` 等校验，当前短写法为 `D`、`I`、`W`、`E`、`F`。
- 日志项包含模块、严重级别、文件名、行号、线程号、时间戳、设备及格式化缓冲区等信息。

## 核心规则

- 为诊断与运维记日志，不写流水账叙述。
- 保留足够的关联上下文便于还原故障。
- 热路径若非收益明确，避免喧哗日志。
- 不要输出密钥、credential、或超大裸负载。
- 不要轻易改变共享日志格式：脚本与运维可能依赖既定形态。

## 建议包含的信息

- 操作或所处阶段；
- 关联实例、连接、租户等可用于关联检索的标识；
- 适用时给出错误码、状态或返回路径；
- 失败可能重试时需带重试上下文。

## 应避免

- 多层重复打出相同语义的一条信息；
- 热路径上对每行数据的闲聊式日志——除非明文要求；
- 会被 grep/告警规则绑定的措辞却频繁漂移；
- 未经更大范围评审就调整严重级别语义、锚点文件名或写入器共用行为。

## 严重级别建议

| 级别 | 含义简述 |
| --- | --- |
| DEBUG | 开发期或深挖排障时用，正常运行不必依赖 |
| INFO | 重要生命周期或可预期状态推进，运维可能希望看到 |
| WARN | 可恢复异常、回退分支或可疑但仍可前进的状态 |
| ERROR | 当前操作对用户或运维可见的失败 |
| FATAL | 不可恢复且往往意味进程终止或同等严重路径 |

## 评审清单（自查）

日志是否可操作？严重程度是否贴切？上下文是否够用且不泄密？是否在热路径带来不必要噪声？共享日志代码改动是否配了更广的验证？
