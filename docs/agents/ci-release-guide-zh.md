# CI 与发布指南

本文为 [`ci-release-guide.md`](ci-release-guide.md) 的中文对照稿。

仓库内具备较强的本地构建与测试入口；目前未必在源码树中挂载单一「托管 Runner」配置文件。应将本文视作**无论你使用 Gitee、Jenkins 还是其他平台**，持续集成与发布自动化应履行的契约。

## 当前验证手段

- 根目录 `Makefile`；
- `qa/run_test_local_v2.sh`、`qa/run_test_v2.sh` 等回归流水线；
- `qa/run_tsbs_test.sh`、`qa/perf_test/*` 等与性能相关工作。

## 当前缺口（维护说明）

编写时若在仓库未见完整托管 CI YAML，则：

- 「CI 拓扑」仍以各团队自建平台为准；
- 「技能化」的失败分类仍可后续补；

（具体条目以英文原版同步为准。）

## 推荐的流水线形态

阶段性顺序（元数据门禁 → 构建 → 聚焦测试 → 共享面拓宽 → 夜间/门禁回归 → 性能闸 → 发布闸），各阶段输入/检查/产出与英文原版表一致——实施时请参考英文表格逐列落地。

## 按改动类型的最低期望示例

与英文原版对齐：触及共享日志/错误、SQL/计划器、TS 引擎/存储、QA 脚手架、性能类改动时应具备的证据形态。

## 推荐工具与实践

归档 CI 产物；Go 可使用 `benchstat` 类对比 TSBS / 火焰图材料；失败后先分类再做「盲重跑」（具体列表见英文版）。

## 发布前清单

外链需求已批准的材料；验证命令与结果；运维可见变更概要；烟雾与监控触点；触发回滚的条件与具体操作。

## 禁止事项

不要盲目重跑失败的 CI；无工作负载论证合入显著性能补丁；不可用 AI 写的 PR 摘要顶替真实验证日志。
