# 错误处理指引

本文为 [`errors.md`](errors.md) 的中文对照稿。

本文约定错误如何在仓库中创建、传播、记录及测试。

## 当前实现事实

- 共享错误基础设施位于 `common/src/error/`。
- `er_cfg.cpp` 维护 `KwdbErrCode` 到人可读元数据及 `PgCode` 的映射。
- `er_error.cpp` 组装具体报错文本，并将若干字段序列化进 JSON。
- `er_stack.cpp` 将错误放入 `KwdbErrorStack`，`DumpToJson()` 导出聚合外形。

## 核心规则

- 尽量保留原始错误码与起因路径；
- 在需要恢复决策、上报或外围诊断时再补充上下文；
- 不要静默吞噬错误；
- 勿随意改写既有编码到文案或到 `PgCode` 的映射；
- 若对外接口要求稳定，则需区分诊断用细节与用户可见的稳定行为。

## 兼容性规则

- 已为某 `KwdbErrCode` 绑定的信息与 `PgCode`，默认视为对用户可见；
- 新增编码时评估是否需要 `PgCode` 以及在 `er_cfg.cpp` 中登记稳定元数据；
- `ToString()` 或 `DumpToJson()` 若改变 JSON 字段形态，应按兼容性改动处理；
- 避免改写下游解析器赖以匹配的措辞。

## 何时添加上下文

- IO 边界、RPC、协议；
- 状态迁移；
- 存储、事务、持久化；
- 对用户可见 SQL 或 API。

## 何时不要堆叠噪声

- 同一失败不要在每一调用层反复包裹；
- 不要用泛化话术覆盖仍有意义的专有码；
- 不要无信息量地多重 log+return。

## 测试期望

- 断言失败语义，不仅是成功路径；
- 若稳定契约依赖分类或编码，断言该契约；
- 若调用者依赖外部 JSON，校验字段形状；
- 除非明文规定「整串消息即契约」，避免脆弱的全字符串比较。

## 评审清单（自查）

原有码链是否仍在？上下文是否增益？在用户可见兼容性上是否退步？失败路径是否与成功路径一视同仁地测过？
