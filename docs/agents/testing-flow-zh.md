# 验证流程

本文为 [`testing-flow.md`](testing-flow.md) 的中文对照稿。

本文约定 agent 与贡献者的默认验证工作流。

## 原则

- 先用最小但可靠的检查证明具体主张。
- 触及共享代码或敏感行为时扩大测试范围。
- 绝不可用更弱的断言替代失败的回归期望。
- 报告成功前应阅读真实输出。

## 常用入口（仓库根执行）

- `make cpplint`
- `make kwdbts2-test`
- `make kwbase-test`
- `make test`
- `make test-logic`
- `make regression-test TEST_NAME=... T='...'`
- `make regression-test-ha`

常用于回归或性能的手工脚本：`qa/run_test_local_v2.sh`、`qa/run_tsbs_test.sh`（参数见英文原版）。

## 验证层级

### 一级：聚焦本地检查

适用于：改动局限在单个文件或窄模块；未改跨模块契约；无已知并发、存储或兼容性风险。
证据示例：针对性单测、包/模块测、必要时局部 lint。

### 二级：模块级检查

适用于：共享模块受影响；模块边界或对外可见行为变化；日志或错误路径变化。
证据示例：模块测试集、靶向回归、相关逻辑测试或 lint。

### 三级：跨模块或敏感检查

适用于：SQL 语义、存储语义、兼容性、并发、性能热点等变化。
证据示例：多级测试组合；必要时附带基准对比或概要。

## 面向常见改动应跑什么

（与英文原版同）：`common` 日志错误、`kwdbts2`、`kwbase` SQL 相关、QA 脚本、构建系统变动的最低验证要求。

## 缺陷修复验证

复现场景尽可能留下记录；自动化可重现时应增加或更新回归；验证失败路径与最近未受影响路径；若涉日志或错误外形须一并核验。

## 性能验证

记录基线命令、数据集、拓扑与环境；优先用仓库自带脚本对比前后；写明方差与样本；若涉及计划变更保留 `EXPLAIN` / `EXPLAIN ANALYZE` 等材料。

## 报告格式

每次变更摘要宜包含：**运行的命令**、**验证覆盖面**、**结果**、**已知缺口**。
